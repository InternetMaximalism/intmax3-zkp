## 0. Executive summary

This report documents a *formal* security audit of INTMAX3 in which the
Plonky2 ZKP circuits **and** the L1 Solidity contracts were transcribed,
line by line, into Lean 4 and either **proved sound** or had their gap
**surfaced as an explicit, machine-checkable finding**. The two halves
are modeled in one framework so the audit can reason **end-to-end** —
from a deposit on L1, through the recursive balance/validity proofs, to a
native ETH withdrawal back on L1 — and state combined-system safety
theorems that neither a circuit-only nor a contract-only review can.

**Bottom line.** No new exploitable vulnerability was found at the circuit
level, the contract level, or the circuit↔contract boundary. The headline
guarantees are now *theorems*:

- **Conservation of funds (global solvency).** Over any sequence of
  contract operations, `Σ ETH withdrawn ≤ Σ ETH deposited`
  (`solvent_from_genesis`). The rollup cannot pay out more than was put in.
- **Backed payouts.** Every native payout is bound to a verified
  `WithdrawalCircuit` proof (`withdrawNative_requires_proof`), which traces
  to a real *sent* transfer (`withdrawal_sound`) backed by a real *spend*
  deduction (`deducts_solvent`).
- **No double-spend / double-withdraw.** Each transfer's nullifier is
  provably one-shot in-circuit (`IndexedMerkle.key_absent`) and consumed
  CEI on-chain (`withdrawLeaf_nullifier_once`).
- **No inflation.** U256 balance arithmetic rejects both overflow and
  underflow in-circuit (`add_no_wrap`, `sub_no_underflow`).
- **Sound anchoring.** A payout's `ext_public_state_commitment` must be a
  finalized state root, written only by a verified validity proof
  (`finalize_only_on_valid`), and finalized roots are permanent
  (`finalized_roots_persist`, `fraud_above_floor`).

**Findings.** One previously-known finding (**F-WITHDRAW-1 = audit622
C-M2**) was *independently re-derived* by the formal model and then
**closed** (the contract re-pins the commitment). One in-circuit finding
(**F-RECIP-1**) was adjudicated **informational** (not fund-exploitable).
All other findings are closed, mostly-resolved, or documented residuals.
No finding remains open as an exploitable risk.

## 1. Methodology

### 1.1 The modeling principle

A Plonky2 circuit is a set of gate constraints over field wires; a proof
exists iff some wire assignment satisfies all of them. A Solidity external
function is a state transition that either commits or reverts. In both
settings, the dangerous bugs are **missing constraints / missing checks** —
a relation the protocol *assumes* but never *enforces*.

We therefore model each unit as a predicate over an arbitrary accepted
execution:

```
-- circuits
Constraints : inputs → outputs → Prop          -- one conjunct per builder.* gate
theorem sound : Constraints i o → nativeSpec i o

-- contracts
f : State → args → Option State                -- none = revert; require/checked-math = guard
theorem invariant : f s a = some s' → P s s'
```

- A provable theorem ⇒ the unit binds/checks what it must.
- An **un**provable strengthening ⇒ the precise missing constraint = the
  finding. Findings are recorded as the strengthening that *fails*, never
  as an admitted (`sorry`) lemma.

Each Lean file cites the exact `source:line` ranges of every constraint it
models, so the transcription is auditable against the implementation.

### 1.2 Trusted base and uninterpreted primitives

- **Algebraic base** (`Zkp/Core/Field.lean`): a commutative field that is
  an integral domain (`mul_eq_zero`). The Goldilocks *characteristic* is
  deliberately **not** axiomatized; the single place it is load-bearing (a
  4-way one-hot in the balance switch board) is surfaced explicitly.
- **Hashes** (Poseidon, keccak) are **uninterpreted functions**:
  determinism is free; collision resistance is a *named* hypothesis
  (`Bytes.PoseidonCR`, `Coverage.KeccakCR`) invoked only where a binding
  relies on it.
- **SNARK/PCS verifiers** (Groth16, KZG, MLE/WHIR) and **signature gates**
  (SPHINCS+/Poseidon) are treated as verified oracles — exactly the audit
  boundary used for cryptographic primitives. Contract verifier wrappers
  (`_verifyMle`, `_verifyKZG`, `BlobKZGVerifier`, `MleVerifier`) are the
  on-chain analog.
- **EVM semantics** (`Zkp/Contracts/Evm.lean`): storage as a record,
  `mapping` as a total function with default, Solidity-0.8 **checked**
  arithmetic as `checkedAdd`/`checkedSub` (revert on over/underflow — this
  revert is frequently the load-bearing invariant), an external call as
  `Option State` (`none` = revert).

### 1.3 Scope

| In scope | Out of scope (uninterpreted / excluded) |
|---|---|
| Balance, withdraw, validity (non-channel) Plonky2 circuits | Poseidon / SPHINCS+ / Regev / MLE-WHIR primitive internals |
| Nullifier indexed-Merkle gadget | Channel-registration circuit internals (channel scope) |
| `IntmaxRollup.sol` (full native fund + optimistic pipeline) | Groth16 / KZG / MLE pairing math (oracles) |
| `ChannelSettlementManager.sol` (payout cap) | `ChannelSettlementVerifier.sol` proof bodies (oracle + nullifier) |

## PART A — ZKP CIRCUITS

### A.1 Architecture recap

INTMAX3 uses four recursive proof families:

1. **Balance** (`src/circuits/balance/`) — per-user private state (spend,
   send-tx, receive-transfer, receive-deposit), routed by a one-hot
   *switch board* and folded by a cyclic *balance circuit*.
2. **Validity** (`src/circuits/validity/`) — block-level consensus: a
   block-hash chain (user-tree updates + block-producer signatures) and a
   deposit-hash chain, bound on-chain via `keccak256(ValidityPublicInputs)`.
3. **Withdraw** (`src/circuits/withdraw/`) — extracts transfers from a
   balance proof and aggregates N withdrawals into one keccak chain.
4. **On-chain wrapper** — MLE/WHIR proof consumed by the L1 contract.

### A.2 Core modeling library (`Zkp/Core/`, 6 modules)

| Module | Pins down |

| `Builder.lean` | gate semantics: `connect`, `assert_*`, `select`, `is_equal`, `not`, `and`, `range_check`, `natLit`, `conditional_assert_eq` |
| `Bytes.lean` | `Bytes32`/`Address`/`HashOut`; Poseidon uninterpreted; `PoseidonCR` named |
| `Merkle.lean` | inclusion gadget (fold + LSB-first index decomposition); conditional verify |
| `U256.lean` | 256-bit values; overflow-rejecting `AddSpec`, underflow-rejecting `SubSpec` |
| `IndexedMerkle.lean` | nullifier non-membership / insert (discharges F-NULL-1) |

  with `is_checked = true` at every in-scope caller — no aliasing.

## PART B — SMART CONTRACTS

### B.1 Architecture recap (`IntmaxRollup.sol`)

An optimistic-rollup settlement pipeline:

1. `deposit()` — escrow native ETH (`totalEscrowed += amount`) and append
   to the deposit hash chain at a sequential `depositCount`.
2. `postBlockAndSubmit()` — post a batch (DA in calldata) with a 1-ETH
   stake, and `_submit()` a keccak commitment over the blob.
3. `finalize()` — on a verified validity proof (`fullVerify`), record the
   state root in `finalizedStateRoots` (permanent) and refund the stake.
4. `fraudProof()` — challenge an invalid/late submission → rollback +
   slash, subject to the rollback floor.
5. `withdrawNative()` / `claimAuthorizedWithdrawal()` — pay native ETH for
   a verified `WithdrawalCircuit` proof (or an authorized burn).
6. `withdraw()` — pull-payment of accrued `pendingWithdrawals`.

### B.2 Contract modeling core (`Zkp/Contracts/Evm.lean`)

External functions are `Option`-returning state transitions; `require` /
checked-math failures are reverts. `checkedSub`/`checkedAdd` model
Solidity-0.8 arithmetic — the `totalEscrowed -= amount` underflow-revert
**is** the global solvency ceiling and is never modeled as saturating ℕ.

### B.4 The proved contract invariants, in words

- **Global solvency** (`solvent_from_genesis`). Modeling deposits and
  withdrawals as a trace, `finalEscrow + Σ withdrawn = initialEscrow + Σ
  deposited`; hence `Σ withdrawn ≤ Σ deposited`. Proven *unconditionally*
  — it does not even need the proofs; the underflow-revert ceiling alone
  guarantees it.
- **Per-call ceiling** (`withdrawNative_solvency`). A single
  `withdrawNative` reduces escrow by exactly the sum paid and reverts if
  that exceeds the escrow held.
- **No double-withdraw** (`withdrawLeaf_nullifier_once` +
  `withdrawLeaf_consumes`). The nullifier is checked then set (CEI) before
  any value movement; a reused nullifier reverts.
- **Payout requires a proof** (`withdrawNative_requires_proof`). Payout is
  gated on (a) MLE/WHIR verification, (b) `pisHash` re-fold binding `ws` to
  the proof, (c) `ext_commitment ∈ finalizedStateRoots`.
- **Sound anchor** (`finalize_only_on_valid`) + **permanent anchor**
  (`finalized_roots_persist`, `fraud_above_floor`). `finalizedStateRoots`
  is written *only* by `finalize` on a verified validity proof, and the
  fraud/rollback machinery can never target a finalized block.
- **Stake conservation & single-resolution.** A 1-ETH bond is refunded or
  slashed exactly once (`spent`/`delete` guard); slash shares sum to the
  bond (`stake_conserved`). No bond is double-paid.
- **Access control.** Burn withdrawals require
  `partialWithdrawalAuthorized`, settable only by a registered settlement
  manager, registrable only by the deployer (two-level).
- **Channel payout cap.** `claimWithdrawalCredit` keeps `totalCreditedOut
  ≤ receivedChannelFunds` — the channel path can never pay out more native
  ETH than it pulled from the rollup (which is itself escrow-bounded).

## PART C — COMBINED-SYSTEM SAFETY

Putting the two halves together yields the end-to-end argument that a
circuit-only or contract-only review cannot make. Every unit of native ETH
that leaves L1:

1. **is backed by a verified proof** — `withdrawNative_requires_proof`
   binds the leaves to a verified `WithdrawalCircuit` proof, which by
   `withdrawal_sound` is a transfer the user genuinely *sent* (present in
   their `sent_tx_tree`, in a settled block), which by `deducts_solvent`
   was covered by a real balance *deduction*;
2. **is anchored to genuine finalized state** —
   `ext_public_state_commitment ∈ finalizedStateRoots`, written only by
   `finalize_only_on_valid` from a validity proof in which
   `signatures_not_skippable` forbids forged (unsigned) blocks, and which
   stays finalized forever (`finalized_roots_persist`);
3. **is paid at most once** — the per-transfer nullifier is
   collision-distinct and one-shot in-circuit (`key_absent`) and consumed
   CEI on-chain (`withdrawLeaf_nullifier_once`);
4. **cannot exceed deposits** — `solvent_from_genesis`: `Σ out ≤ Σ in`,
   independent of the proofs;
5. **(burn path) requires channel consent** — gated by a settlement-manager
   authorization binding all fields, and capped on the channel side by
   `claim_preserves_cap`.

**F-WITHDRAW-1 closed in the model.** The circuit leaves 5 extended-state
fields as free witnesses, yet the contract requires the *full* commitment
to be a finalized root. Since finalized roots come only from validity
proofs that constrain those 5 fields, collision resistance forces the
prover to use the genuine tuple. The circuit's partial binding is
intentional and **completed contract-side** — exactly the link the
combined model makes explicit (`finalize_only_on_valid` +
`withdrawNative_requires_proof` + `KeccakCR`).

| **F-NULL-1** | — | **Discharged** | Spend-once depends on the nullifier non-membership proof. `IndexedMerkle.key_absent` proves the bracketing (`low.key < key < low.next_key` + low-leaf inclusion) implies absence; `empty_leaf_cannot_be_low` (key=MAX sentinel) closes the documented `nullifier_duplicate_insertion_poc`. |

| **F-SPEND-1** | — | **Closed** | `spend_circuit` computes `is_valid` but does not assert it. `send_tx` consumes it as a no-op selector (`invalid_spend_is_noop`) and `receive_transfer` asserts it true — so an invalid spend cannot advance or be received. Not a vuln. |
| **F-ACCT-1** | — | **Closed** | `is_checked`-gated index range checks; all non-test callers pass `true`, and index widths match tree heights ⇒ no aliasing. |
| **F-BLKR-1** | Low | **Mostly resolved** | `block_r ≤ block_number` invariant: enforced on the receive paths (`blockR_bounded`); the send path bounds `block_r` by ordering against `tx_block_number`. |
| **F-AUX-1** | — | **Residual (by design)** | Inter-channel `aux_data == tx_leaf_hash` is enforced off-circuit at co-sign time (documented SECURITY comment, threat model F3-A), not in-circuit. The chain fold itself is faithful; the equality is a co-signer trust assumption. |

No finding remains open as an exploitable risk.

## PART E — TRUSTED ASSUMPTIONS & LIMITATIONS

The audit's conclusions are conditional on the following, each of which is
either standard, named in the model, or out of the stated scope:

1. **Cryptographic primitives are sound.** Poseidon/keccak collision
   resistance (`PoseidonCR`/`KeccakCR`), and the MLE/WHIR, KZG, Groth16,
   and SPHINCS+/Poseidon-signature verifiers, are trusted oracles. Their
   internal soundness is out of scope (and is the subject of separate
   review — e.g. the WHIR on-chain verifier work).
2. **Layout equality (circuit ↔ contract).** The keccak/Poseidon
   preimage layouts on the contract side are byte-identical to the Rust /
   in-circuit ones. This is asserted by the repo's differential tests
   (e.g. `test_channelRegPreimageDifferential`,
   `test_withdrawalLeafDifferential`) and taken as a modeling assumption.
3. **The validity circuit constrains the 5 extended-state fields.** The
   F-WITHDRAW-1 closure depends on this; it is the validity circuit's own
   invariant (D3/A8/R4), documented in code and audit622, and partially
   modeled here (`signatures_not_skippable`, `bp_sig_chain_threaded`,
   `sequential_append`).
4. **Channel scope.** Channel-registration circuit internals and the
   `ChannelSettlementVerifier` proof bodies are uninterpreted; the channel
   *fund* safety is captured by the manager payout cap, not the close/claim
   proof semantics.
5. **Deployment.** Correct VK deployment, block-producer / settlement-
   manager key custody, and L1 censorship/availability are operational
   concerns outside this audit.
6. **Goldilocks characteristic** is relied upon in exactly one place (the
   4-way one-hot); it is surfaced, not hidden.

This is a *relative* soundness result: under the trusted base, the modeled
logic is correct. It does not replace cryptographic review of the
primitives, nor dynamic testing of the deployed bytecode.

## PART G — CONCLUSION

INTMAX3's ZKP circuits and L1 contracts were formalized in Lean 4 and
machine-checked end to end. Under the stated trusted base, the system's
core fund-safety properties — **conservation (Σ out ≤ Σ in)**, **backed and
single-use payouts**, **no inflation**, **sound and permanent anchoring**,
and **stake/channel-cap solvency** — hold as theorems, and the
circuit↔contract boundary (F-WITHDRAW-1 / C-M2) is closed inside the model.
No new exploitable vulnerability was found. The remaining items are an
informational encoding nit (F-RECIP-1) and a documented off-circuit
co-sign assumption (F-AUX-1).
